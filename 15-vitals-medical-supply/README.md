# VITALS — DHA RESCUE Blood Logistics

**Joint Blood Logistics Decision Support for USINDOPACOM Distributed Maritime
Operations.** App **#15 / 20** — MDM 2026 LOGCOM AI Forum Hackathon, Phase 2.

> "One depot, twelve spokes, sixty hours of sustainment. VITALS predicts which
> spoke fails first."

## What it does

A hub-and-spoke decision-support agent for blood and blood-product logistics
across one regional medical depot (Apra, Guam) and twelve distributed Marine
spoke nodes (afloat L-class amphibs, a CVN BAS, expeditionary Role-2 sites,
EABO Role-1 forward sites, and a marine corps air station clinic).

1. **Ingests** synthetic Medical Supply Inventory and Medical Supply Network
   Data Model rows (PRBC / PLASMA / PLT / LTOWB units, expirations, cold-chain
   status, daily consumption, days-of-supply per site; hub<->spoke route status,
   lift availability, casualty assumptions, approved buy-on-market vendor list).
2. **Scores** all 12 spokes 0–10 with a structured-JSON LLM call
   (`shared.kamiwaza_client.chat_json`). Schema:
   `{node_id, days_of_supply, projected_stockout_date, viability_index,
     top_constraint, confidence}`.
   Deterministic `baseline_scores()` fallback always renders a non-empty map.
3. **Generates** a one-page **Commander's Decision Brief** in markdown with
   BLUF + top-3 spoke nodes at risk + secondary-effect cascades (cold-chain
   loss, transport delays, lab-reagent shortage) + 3–5 recommended mitigation
   actions citing approved vendors by id (V-DRYICE-01, V-DONOR-04, etc.). Hero
   call uses the Kamiwaza-deployed hero model with a 35 s wall-clock timeout.
4. **Renders** a Folium hub-and-spoke map (CartoDB dark) with viability-colored
   spoke markers and lift-status-styled legs (solid GREEN, dashed AMBER, dotted
   RED), plus a ranked viability list, scenario-adjusted inventory table, and
   the approved vendors expander.

### Hero AI move (two-step)

```
Step 1   chat_json  → strict JSON, 12 spoke scores + viability index
Step 2   chat       → "gpt-5.4" hero, 35s wall-clock, deterministic fallback
```

Both steps are wrapped in `concurrent.futures.ThreadPoolExecutor` with hard
timeouts (`HERO_CALL_TIMEOUT_S=35`, `SCORING_CALL_TIMEOUT_S=25`). Briefs for
the four built-in scenarios are pre-computed by `data/generate.py` into
`data/cached_briefs.json` so the demo recording never blocks on a live LLM call
— the live regenerate button still re-runs in real time.

## Mission frame

Maps to the **DHA RESCUE** use case (DHA / NAVMED logistics decision support)
on the LOGCOM portal for MDM 2026, plus the umbrella problem set in
**MARADMIN 131/26**: *contested logistics, supply chain management,
expeditionary operations*. Stand-In Forces deployed across the First and Second
Island Chains depend on this exact distribution pattern.

The Commander's Decision Brief format (BLUF + cascades + actions) is the
language the J-4 / G-4 medical-logistics cell already uses — the brief is
print-ready and matches the standing daily.

## Run it

```bash
# 1. install deps (once)
python3 -m pip install -r requirements.txt

# 2. generate synthetic data + pre-compute hero briefs
python3 data/generate.py

# 3. start the FastAPI backend on 8015
python3 -m uvicorn backend.app:app --host 0.0.0.0 --port 8015

# 4. start the Streamlit frontend on 3015
streamlit run src/app.py \
  --server.port 3015 \
  --server.headless true \
  --server.runOnSave false \
  --server.fileWatcherType none \
  --browser.gatherUsageStats false
```

Open http://localhost:3015. Pick a scenario in the sidebar (try **Hub airlift
loss**), then click **CONSTRAIN HUB & GENERATE BRIEF**.

## On-prem swap (Kamiwaza)

Set the two env vars and the same code runs against a vLLM endpoint inside the
accredited environment — no app changes:

```bash
KAMIWAZA_BASE_URL=https://kamiwaza.local/api/v1
KAMIWAZA_API_KEY=...
```

The shared multi-provider client (`shared/kamiwaza_client.py`) auto-detects the
provider; everything from the structured scoring step to the hero brief routes
through Kamiwaza's OpenAI-compatible surface. Air-gapped, IL5/IL6 ready.
